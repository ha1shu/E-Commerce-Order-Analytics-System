## E-Commerce Order Analytics System

An end-to-end data pipeline built with **Python** and **SQLite** that generates messy e-commerce data, cleans it, loads it into a relational database, and produces business analytics through SQL and a command-line reporting tool.

This project was built as a mini-project to practice real-world data engineering skills: handling dirty data, writing clean pandas transformations, and using intermediate-to-advanced SQL (window functions, CTEs, cohort analysis) to answer business questions.

---

### Project Overview

Online order data rarely arrives clean. This project simulates that reality:

1. **Generate** realistic but intentionally messy order data (missing IDs, bad date formats, invalid emails, negative quantities, broken referential integrity).
2. **Clean** the data using pandas, logging every issue found along the way instead of silently fixing it.
3. **Load** the cleaned data into a local SQLite database.
4. **Analyze** it using 16 SQL queries, from basic aggregations to advanced window functions and cohort retention analysis.
5. **Report** on it through a command-line tool that takes a date range and prints a business summary.
6. **Test** the pipeline against edge cases (bad order references, invalid discounts, zero quantities, future-dated orders) to confirm the cleaning logic behaves as intended.

### How to Run (in order)

```bash
# 1. Generate raw, messy data
python python_script/generation_data.py

# 2. Clean the data and produce the issues report
python python_script/clean_data.py

# 3. Load cleaned data into SQLite
python load_to_sqlite.py

# 4. Run all 16 SQL queries and export results as CSVs
python run_queries.py

# 5. Generate an on-demand business report (interactive prompts)
python python_script/report_cli.py

# 6. Run edge case tests
python TestEdge_Cases.py
```

Each step depends on the one before it, so they should be run in this order the first time. After that, any step can be re-run independently (e.g. re-running `report_cli.py` alone to pull a different date range).

### Data Quality Issues (Intentionally Introduced & Detected)

The raw data was generated with the following deliberate issues, to simulate a real messy production dataset:

| Issue | Where | Handling |
|---|---|---|
| Missing `customer_id` | `orders.csv` (~5%) | Flagged, excluded from customer-level aggregations |
| Inconsistent date format (`DD-MM-YYYY` vs `YYYY-MM-DD HH:MM:SS`) | `orders.csv` | Parsed and normalized to a single datetime format |
| Invalid emails (missing `@` or domain) | `customers.csv` (~2%) | Detected via regex validation, listed in issues report |
| Negative quantity (returns) | `order_items.csv` (~3%) | Kept intentionally — used to calculate return rates |
| `discount_percent` outside 0–100 | `order_items.csv` | Flagged as invalid, excluded from revenue calculations |
| Messy product names (extra spaces, mixed case) | `products.csv` | Normalized via trim + title case |
| `order_items` referencing non-existent `order_id` | `order_items.csv` | Detected via referential integrity check, logged and excluded |

The exact counts of each issue found are written to `cleaned_data/issues_report.txt` after running `clean_data.py`.

### SQL Analysis — Query Summary

All 16 queries live in `sql_script/analysis_query.sql`, grouped into three tiers:

**Basic**
1. Total revenue per category
2. Top 10 customers by total order value
3. Month-wise order count (last 12 months)

**Intermediate**
4. Customers who ordered but never had anything delivered
5. Products with more returns than purchases
6. Return rate per category

**Advanced (Window Functions, CTEs, Subqueries)**
7. Running total of revenue per region (`SUM() OVER`)
8. Product ranking within category (`DENSE_RANK()`)
9. Days between consecutive orders per customer, with an "At Risk" flag (`LAG()`)
10. Multi-level CTE: monthly revenue → spend category → count per month
11. Customer segmentation into quartiles (`NTILE(4)`)
12. Year-over-year revenue comparison
13. First vs. most recent purchased category per customer, with a category-shift flag
14. Cumulative revenue distribution (what % of revenue comes from top customers)
15. Cohort analysis by registration month, with month 0–3 retention rates
16. Self-join to find frequently bought-together product pairs

Results of each query are exported individually as CSVs into `query_results/` by `run_queries.py`, for easy review without needing to re-run SQL manually.

### Command-Line Report Tool

`report_cli.py` is an interactive tool that:
- Accepts a report type (`daily` / `weekly` / `monthly`)
- Accepts a custom date range
- Returns: total orders, total revenue, unique customers, and top 3 products for that period
- Automatically compares the period against the immediately preceding period of equal length, showing % revenue change

This was built using only the standard `sqlite3` library (no external dependencies), per the project requirements.

---

### Edge Case Testing

`TestEdge_Cases.py` verifies the pipeline's behavior against four scenarios that real order data can produce:

1. `order_items` referencing an `order_id` that doesn't exist in `orders`
2. `discount_percent` greater than 100 (or negative)
3. `quantity` equal to 0
4. `order_date` set in the future

Each test asserts the row is correctly detected, and documents the business decision behind how it should be handled (flagged vs. excluded vs. capped) rather than just checking the code doesn't crash.

### Tech Stack

- **pandas** — data cleaning and transformation
- **Faker** — realistic fake data generation
- **SQLite** — local relational database
- **SQL** — window functions, CTEs, subqueries for analysis

---


- Designing data with intentional flaws (rather than clean synthetic data) forces more realistic cleaning logic and referential integrity checks.
- Window functions (`LAG`, `NTILE`, `DENSE_RANK`) and multi-level CTEs are far more readable and maintainable than equivalent logic written in application code.
- Logging data issues instead of silently dropping them makes the pipeline auditable — a decision that matters more in production systems than in one-off scripts.
